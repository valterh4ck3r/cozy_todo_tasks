FOLDERS.md

This document explains the purpose and typical structure of folders found in this repository. It is intended to help contributors quickly find where source, resources, build outputs, and configuration live.

Top-level layout

- `androidApp/`
  - Purpose: Android application module. Contains Android-specific app code and resources.
  - Common files and folders:
    - `build.gradle.kts` - module Gradle build script (Kotlin DSL).
    - `src/` - source sets for the Android app.
      - `main/` - main Android application source (Java/Kotlin, AndroidManifest.xml, resources under `res/`).
    - `build/` - Gradle build outputs (compiled classes, intermediates, generated resources).
    - `kotlin/`, `outputs/`, `generated/` (subfolders created by the build process).

- `desktopApp/`
  - Purpose: Desktop (JVM) application module.
  - Common files and folders:
    - `build.gradle.kts` - module build script.
    - `src/main/` - main JVM application source.
    - `build/` - build artifacts and class files for the desktop app.

- `iosApp/`
  - Purpose: iOS application module (Xcode project + Swift UI code).
  - Common files and folders:
    - `iosApp.xcodeproj/` - Xcode project files.
    - `iosApp/` - Swift source files, assets, Info.plist, and supporting resources.
    - `Configuration/` - Xcode configuration files (e.g. `Config.xcconfig`).

- `sharedLogic/`
  - Purpose: Kotlin Multiplatform module containing shared business logic and platform-specific adapters.
  - Common files and folders:
    - `build.gradle.kts` - module build script.
    - `src/` - multiplatform source sets:
      - `commonMain/` - shared code across platforms (Kotlin).
      - `commonTest/` - shared tests.
      - `androidMain/`, `jvmMain/`, `iosMain/` - platform-specific implementations.
    - `build/` - compilation outputs and intermediates (e.g., Kotlin/Native artifacts).

- `sharedUI/`
  - Purpose: Kotlin Multiplatform UI module (shared UI components, theming).
  - Common files and folders:
    - `build.gradle.kts` - module build script.
    - `src/commonMain/` - shared UI code.
    - `src/commonTest/` - tests for shared UI components.
    - `build/` - build outputs.

- `build.gradle.kts`, `settings.gradle.kts`, `gradle.properties`, `gradle/`, `gradlew`, `gradlew.bat`
  - Purpose: Project-level Gradle build configuration and wrappers.
  - `gradle/` contains wrapper and local gradle configuration like `libs.versions.toml`.
  - `gradlew`/`gradlew.bat` are the Gradle wrapper scripts for Unix/Windows.

- `build/` (top-level)
  - Purpose: Top-level build reports and project-wide generated outputs.
  - Contains reports and caches created by Gradle.

- `docs/`
  - Purpose: Documentation for the project.
  - Subfolders (example): `en_US/` and `pt_BR/` containing architecture docs and images.

- `local.properties`
  - Purpose: Local development configuration (usually includes SDK paths). Not committed to all repositories.

- `README.md`
  - Purpose: Project introduction and basic instructions.

Common patterns inside each module

- `build.gradle.kts` (or `build.gradle`) - Every module normally has a build script at its root.
- `src/` - Source tree. For Kotlin Multiplatform modules this commonly contains nested folders for platform-specific and common source sets: `commonMain`, `androidMain`, `jvmMain`, `iosMain`, plus corresponding `*Test` folders.
- `src/main/` - For single-platform modules (Android, JVM), typical location for application code, resources and manifests.
- `build/` - Generated by Gradle; contains compiled classes, generated resources, intermediate files, packaging outputs, and module reports. This folder is build-generated and usually ignored by version control.
- `generated/` - Subfolder for generated sources (codegen, kapt, compose compiler outputs, etc.).

Notes about artifacts and build outputs

- Many folders under `build/` are managed by Gradle and used during incremental builds (e.g., `intermediates/`, `outputs/`, `kotlin/`). Do not manually edit files inside `build/`.
- If you need to inspect generated sources or resources, check the `build/generated/` and `build/intermediates/` folders in the relevant module.

Where to add new code

- Feature/UI code that should be reused across platforms should go into `sharedLogic/src/commonMain` or `sharedUI/src/commonMain` so the code is visible to all targets.
- Platform-specific implementations should live in the platform source sets (e.g., `androidMain`, `iosMain`, `jvmMain`).
- App-specific wiring (activities, app manifests, entry points) belongs to `androidApp/src/main`, `iosApp/iosApp` (Swift), and `desktopApp/src/main` respectively.

Version control and housekeeping

- Keep build outputs out of version control. The `build/` folders are generated and should be ignored.
- Keep platform configuration (e.g., `Config.xcconfig`) in their respective platform modules.

Quick reference (paths shown relative to repository root)

- `androidApp/` - Android app sources and resources
- `desktopApp/` - Desktop (JVM) app
- `iosApp/` - iOS app and Xcode project
- `sharedLogic/` - Cross-platform business logic (Kotlin Multiplatform)
- `sharedUI/` - Cross-platform UI components
- `gradle/`, `gradlew`, `gradlew.bat` - Gradle wrapper and config
- `docs/` - Documentation
